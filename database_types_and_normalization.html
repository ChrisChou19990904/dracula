資料庫種類與功能說明

1. 關聯式資料庫 (Relational Database)

以表格形式存資料，強調資料一致性 (ACID)。適合金融、訂單、ERP 系統。

資料庫 功能/特點
MySQL 開源、支援 SQL 查詢、適合中小型網站
PostgreSQL 支援複雜查詢、ACID、擴展性強
Oracle Database 企業級關聯式資料庫，強一致性、穩定可靠
Microsoft SQL Server 企業應用、整合微軟生態系統

2. 分散式 / NoSQL 資料庫

資料分散在多台伺服器，支援高並發與水平擴展，部分資料庫採用最終一致性。

資料庫 功能/特點
MongoDB 文件型資料庫 (Document DB)，適合存 JSON 結構資料
Redis 快取資料庫，支援即時訊息、排行榜、計數器
Cassandra 高可用、分散式，適合大數據與高流量應用
DynamoDB AWS 提供的 NoSQL，易水平擴展、免維護伺服器
TiDB 分散式關聯式資料庫，支援 SQL 與水平擴展

3. 資料庫使用建議

  • 純單機遊戲：可以完全不使用資料庫，存檔在本地即可。
  • 多人遊戲或線上服務：SQL 用於帳號、課金；NoSQL 用於即時狀態、快取、排行榜。
  • 大型遊戲：混合架構最常見（SQL + Redis + 專用遊戲服務器）。

4. 正規化 (Database Normalization)

正規化是整理資料表結構的方法，目的是減少資料冗餘與避免異常更新。

  • 第一正規化 (1NF)：每個欄位都是原子值，表格中不能有重複欄位或集合。
  • 第二正規化 (2NF)：滿足 1NF，且每個非主鍵欄位完全依賴主鍵。
  • 第三正規化 (3NF)：滿足 2NF，且非主鍵欄位不依賴其他非主鍵欄位。

5. 正規化 SQL 範例

初始表 (未正規化)

CREATE TABLE Players ( PlayerID INT, PlayerName VARCHAR(50), Game1 VARCHAR(50), Game1Score INT, Game2 VARCHAR(50), Game2Score INT );

1NF - 每個欄位都是原子值

CREATE TABLE PlayerGames ( PlayerID INT, PlayerName VARCHAR(50), GameName VARCHAR(50), Score INT, PRIMARY KEY (PlayerID, GameName) );

2NF - 消除部分依賴，拆表

CREATE TABLE Players ( PlayerID INT PRIMARY KEY, PlayerName VARCHAR(50), CountryID INT ); CREATE TABLE Countries ( CountryID INT PRIMARY KEY, CountryName VARCHAR(50) ); CREATE TABLE PlayerGames ( PlayerID INT, GameName VARCHAR(50), Score INT, PRIMARY KEY (PlayerID, GameName) );

3NF - 消除傳遞依賴

Countries 表格讓 Player 表不再依賴非主鍵欄位 (CountryName)，符合 3NF。

6. 總結

  • 正規化可以避免資料重複、減少錯誤、便於維護。
  • 大部分實務應用到 3NF 就足夠。
  • 資料庫種類不同，選擇要依遊戲需求或系統需求決定。
